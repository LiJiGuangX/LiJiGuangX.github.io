# 导师端（mobile2）代码现状分析与建议文档

**目的**：梳理 mobile2 目录中的实现代码，对标规格文档，识别功能缺口和决策点。

---

## 1. 等级系统 — 当前设计与建议

### 1.1 现有代码现状

**config.js：**
```javascript
const TUTOR_LEVELS = [
  { lv: 1, name: '初级', icon: '📝', ... },
  { lv: 2, name: '中级', ... },
  { lv: 3, name: '高级', ... },
  // 已删除 Lv4 资深、Lv5 金牌
]
```

**tutor.html：**
- Lv 硬编码为 `CUR_LV = 3`（当前用户等级）
- 等级展示：在 `initLevels()` 中动态渲染等级总览
- 等级徽章：用于导师卡片预览、我的页面等处展示

**UI中的等级展现位置：**
1. 咨询师信息设置 → 预览卡片
2. 时间管理 → 费率卡（显示当前等级和推荐价格）
3. 我的 → 导师头部信息栏
4. 等级成长页（**当前未实现**）

---

### 1.2 我的观察与建议

**问题 1：Lv 级数决策**

目前删除了 Lv4（资深）和 Lv5（金牌），改为 **3 级制**（初级/中级/高级）。

| 选项 | 优点 | 缺点 |
|------|------|------|
| **保持 3 级** | 导师易抵达高级等级，激励效果强；代码改动最小 | 等级跨度小，高端导师差异化不足 |
| **恢复 5 级** | 差异化明显，高端导师有目标感；符合原设计 | 导师难抵达高级，可能打击积极性 |
| **折中：4 级**（建议） | 平衡激励和差异化；Lv1初级→Lv2中级→Lv3高级→Lv4资深 | 需补充 Lv4 的参数 |

**建议**：确认最终要几级？ 这决定了 config.js 中 TUTOR_LEVELS 的长度。

---

**问题 2：价格系数与实现**

目前 config.js 中 Lv1 有具体数值，但 Lv2/Lv3 都是 `null`：
```javascript
Lv1: { priceCoeff: 1.0, exposureWeight: 1.0, commissionBonus: 0 }
Lv2: { priceCoeff: null, exposureWeight: null, commissionBonus: null }
```

**现状**：时间管理页的费率卡是 hardcode 的示例数据（¥30, ¥80, ¥130, ¥180），没有根据等级显示。

**建议**：需要产品定义：
- Lv1 初级的建议价格是多少？（当前规格说 50-100元）
- Lv2、Lv3 的价格系数和建议范围
- 这些参数应该写入 config.js，然后在代码中读取

---

**问题 3：KPI 指标与升级条件**

规格文档中说升级需要同时满足 6 个条件（成长值 + 响应率 + 完成率 + 好评率 + 累计次数 + 无违规）。

**现状**：代码中完全没有实现 KPI 计算、成长值累积、升级判断等逻辑。只有静态页面展示。

**建议**：这些都需要后端支持和前端 JS 逻辑。需要：
- 从后端 API 获取导师当前的各项 KPI 数值
- 前端显示进度条和达标状态
- 升级发生时触发通知

---

### 1.3 等级相关的 UI 缺口

**已实现的：**
- ✅ 等级显示（徽章、名称、图标）
- ✅ 等级总览列表（在等级成长页中）
- ✅ 预览卡片中的等级徽章

**缺口：**
- ❌ 等级成长页的详细实现（当前只有 HTML 框架，无 JS 逻辑）
  - 成长值进度条
  - 升级条件列表（✓/✗ 标记）
  - 升级按钮
  - 福利说明（专场直播、佣金加成等）

## 2. 学历与认证 — 「与学历无关」的实现现状

### 2.1 代码现状

**注册认证页面（p-reg）：**
```html
<input class="fi" placeholder="请输入or选择院校名称">  <!-- 毕业院校 -->
<input class="fi" placeholder="请输入or选择专业名称">   <!-- 所学专业 -->
<input class="fi" placeholder="请输入学信网验证码">       <!-- 学信网验证 -->
```

**我的页面：**
```html
<div class="mine-meta">北京大学 · 计算机科学 &nbsp;·&nbsp; <span class="bdg" id="mine-badge">...</span></div>
```

**信息设置页预览卡片：**
```html
<div style="font-size:12px;color:#6b7280;margin-top:4px">北京大学 · 计算机科学</div>
```

---

### 2.2 我的观察

**学历信息现状：**
1. 在认证页面 **主动采集** 了学历信息
2. 学历信息在多个地方 **显示给导师和用户**
3. 规格文档说"学历用于用户信任评估，不影响等级"——但代码没有实现"等级与学历脱钩"的验证逻辑

**问题：**
- 认证流程没有检查 ❌"学历验证是否与等级绑定"
- 代码中硬编码了特定导师的学历（"北京大学·计算机科学"），不够灵活

**我的理解：**
> "与学历无关" = 导师的初始等级永远是 Lv1，无论学历是高中、本科还是博士。

**建议**：明确规则——
- 学历信息应该在 **用户端** 展示成额外的信任指标（"来自北大"），但
- **不应该** 影响工作端显示的等级、价建议、收手续费

## 3. 预约管理 — 现有功能完整度

### 3.1 代码现状

**已实现的功能：**
- ✅ 预约列表（待确认、已确认、已完成、已取消）
- ✅ 搜索和日期筛选
- ✅ 待确认状态的 10 分钟倒计时（自动变色）
- ✅ 接受预约 → 确认
- ✅ 取消预约 → 弹窗选择原因
- ✅ 预约分组显示（按档期）

**数据结构：**
```javascript
const APPTS = [
  { id, status, date, timeRange, name, question, deadline: _now + ...ms }
]
```

**状态流转支持：**
- pending → confirmed（接受）
- pending → cancelled（拒绝）
- confirmed → cancelled（违约）
- completed（自动，时间到期）

---

### 3.2 我的观察

**优点：**
1. 时间倒计时实现完善（每秒更新、颜色梯度变化）
2. 已有完整的预约数据模型
3. 取消原因选项齐全

**缺口：**
1. **没有与后端 API 对接** — APPTS 是 hardcode 的示例数据
2. **没有实现的状态：**
   - 超时未确认状态处理（当 deadline 过期）
   - 进行中状态（视频会议）
   - 完成后的评价反馈机制
3. **没有实现的处罚规则：**
   - 导师超时未确认 → 自动取消 + 成长值扣分
   - 导师违约取消 → 扣手续费
   - 响应率指标计算

---

### 3.3 建议

**优先级 1（高）：**
- 连接后端 API，替换 hardcode 数据
- 实现超时自动取消逻辑

**优先级 2（中）：**
- 实现进行中状态和视频会议按钮
- 实现成长值、响应率的后端计算

**优先级 3（低）：**
- 导师端显示预约笔记内容

---

## 4. 时间管理 — 日历和时间段

### 4.1 代码现状

**已实现的功能：**
- ✅ 月份日历视图（上个月/下个月切换）
- ✅ 日期高亮（有时间段的日期显示不同样式）
- ✅ 点击日期显示该日期的时间段列表
- ✅ 时间段添加表单（小时、分钟、时长、循环、月份）
- ✅ 时间段删除
- ✅ 时间图谱（周视图，显示某周各天的时间占用）

**数据结构：**
```javascript
const SS = {
  slots: [
    { id, hour, minute, duration, weekdays: ['一','二',...], months: ['3月',...], date: null }
  ]
}
```

---

### 4.2 我的观察

**完整度评分：✅ 80%**

**优点：**
1. 交互逻辑完善（日历高亮、时间段管理）
2. 数据模型合理（支持单日和循环）
3. 时间图谱可视化做得好

**缺口：**
1. **没有后端对接** — 时间段保存是本地演示
2. **费率卡是静态的** — 显示的费用没有根据等级和时长动态计算
3. **没有冲突检测** — 添加时间段时，缺少"是否已有冲突"的验证

| 决策项 | 当前状态 | 建议 | 优先级 |
